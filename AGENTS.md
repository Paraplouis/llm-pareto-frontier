# Repository Guidelines

## Project Structure & Module Organization

This repository is a static D3 visualization with Python data utilities. The app entry point is `index.html`, styles live in `styles.css`, and browser code is under `js/`: `app.js` orchestrates modules in `js/modules/` (`dataProcessor.js`, `chartRenderer.js`, `uiController.js`). Data files are in `data/`; commit the deployable outputs `synthesized_data.js` and `price_data.json`, while raw/debug refresh artifacts are ignored. Python data scripts live in `utils/`. Tests are in `tests/`, split between Node `.mjs` tests and Python `pytest` tests.

## Build, Test, and Development Commands

- `python -m http.server`: serve the site locally at `http://localhost:8000`.
- `python -m pip install -r requirements.txt`: install data-refresh dependencies when present.
- `./refresh.sh`: fetch OpenRouter pricing, fetch LM Arena rankings from Hugging Face, and regenerate synthesized data.
- `python -m pytest`: run Python tests for `utils/`.
- `node --test tests/test_dataProcessor.mjs`: run the Node unit tests for data-processing logic.
- `node tests/test_browser.mjs`: run the Puppeteer smoke test after starting the local server; requires Puppeteer and Google Chrome.

## Coding Style & Naming Conventions

Use 4-space indentation in JavaScript and Python. JavaScript uses ES modules, `PascalCase` for classes, and `camelCase` for methods and variables. Python uses `snake_case`, type hints where practical, dataclasses for structured records, and `pathlib` for paths. Keep model-name normalization logic synchronized between `js/modules/dataProcessor.js` and `utils/generate_synthesized_data.py` when changing matching behavior.

## Testing Guidelines

Add focused tests beside the related suite in `tests/`. Name Python tests `test_*.py` and Node tests `test_*.mjs`. For data matching changes, cover both the Python pipeline and the mirrored JavaScript behavior. For UI or rendering changes, run the browser smoke test and inspect the generated `tests/screenshot_test.png` if layout behavior changed.

## Commit & Pull Request Guidelines

Recent commits use concise Conventional Commit-style prefixes such as `fix:`, `test:`, `refactor:`, and `chore:`. Keep subjects imperative and specific, for example `fix: sync model normalization rules`. Pull requests should describe the user-visible change, list commands run, note data files regenerated by `./refresh.sh`, and include screenshots for chart or layout changes. Link related issues when applicable.

## Security & Configuration Tips

Do not commit `.venv/`, `node_modules/`, raw refresh outputs, or debug pricing files. Treat external pricing and leaderboard data as untrusted input; preserve escaping and sanitization around tooltips and rendered model metadata.
